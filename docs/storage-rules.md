# Yeti Cloud — Правила хранения данных (Storage Rules)

## 1. Общие принципы

- Все устройства синхронизируют **одну и ту же папку** `Documents/YetiCloud`
- Сервер выступает **транзитным узлом**: хранит файлы временно, пока целевое устройство не в сети
- Сервер **доверенный**, монопольно выделен пользователю (в будущем — изолированные среды для разных пользователей)
- Файлы на сервере хранятся в открытом виде (шифрование на клиенте — опционально, в будущем)

## 2. Структура папок на устройствах

### Windows
```
%USERPROFILE%\Documents\YetiCloud\
├── ...                      # Файлы пользователя
├── .yeti\                   # Служебная папка клиента
│   ├── index.json           # Индекс файлов (пути + хеши + время изменения)
│   ├── state.json           # Состояние синхронизации (последний sync с каждым устройством)
│   └── pending\             # Локальные изменения, ожидающие отправки
│       └── ...
├── .services\               # Обменные файлы приложений-сателлитов
│   ├── notes\               # Сервис заметок
│   │   └── notes.db         # База данных заметок (SQLite)
│   ├── finance\             # Финансовый инструмент
│   │   └── finance.db       # База данных финансов (SQLite)
│   └── ...
└── .conflicts\              # Конфликтные версии файлов
    └── ...
```

### Android
```
/storage/emulated/0/Documents/YetiCloud/
├── ...                      # Файлы пользователя
├── .yeti\                   # Служебная папка клиента
├── .services\               # Обменные файлы приложений-сателлитов
└── .conflicts\              # Конфликтные версии файлов
```

> **Примечание**: на Android используется SAF (Storage Access Framework) для доступа к `Documents`. Приложения-сателлиты запрашивают доступ к `Documents/YetiCloud/.services/<имя_сервиса>` через SAF.

## 3. Структура staging на сервере

```
/opt/yeti-server/staging/
├── <device_id>\
│   ├── pending\              # Файлы, ожидающие доставки на устройство
│   │   ├── photo.jpg         # Сам файл
│   │   └── photo.jpg.meta    # Метаданные: исходный хеш, от кого, когда создан, размер
│   └── conflicts\            # Конфликтные версии (если файл изменён на двух устройствах)
│       └── .c202606032139_a1b2c3\
│           └── sheet.xlsx    # Версия файла от устройства a1b2c3 на момент 2026-06-03 21:39
```

### Формат `.meta` файла
```json
{
  "original_hash": "sha256...",
  "source_device": "device_id",
  "target_device": "device_id",
  "created_at": "2026-06-03T21:39:00Z",
  "size": 1048576,
  "priority": "normal"
}
```

## 4. Политика хранения на сервере

| Правило | Описание |
|---|---|
| **Очередь** | Файлы для офлайн-устройства хранятся в `staging/<device_id>/pending/` до момента доставки |
| **Переполнение** | При нехватке места на сервере файлы удаляются, остаётся только `.meta`-заглушка с пометкой `"evicted": true`. Устройство при подключении видит уведомление и запрашивает файл у источника |
| **Конфликт** | Если файл изменён на двух устройствах, пока третье было офлайн — сервер сохраняет обе версии в `conflicts/.c<timestamp>_<device_id>/`, а устройству присылает уведомление о конфликте |
| **Очистка** | Доставленные файлы удаляются сразу. Конфликтные папки старше 30 дней удаляются автоматически |

## 5. Папка `.services`

Назначение: изолированный обмен данными между приложениями-сателлитами (заметки, финансы и др.).

Правила:
- Каждый сервис создаёт свою подпапку: `.services/<имя_сервиса>/`
- Сервис отвечает за формат и целостность своих файлов
- Рекомендуемый формат баз данных: **SQLite** (хорошо переносит файловую синхронизацию, не требует сервера)
- Сервис **не должен** писать в чужие подпапки
- Yeti Cloud синхронизирует `.services` наравне с обычными файлами, но не анализирует содержимое

Примеры:
| Сервис | Подпапка | Файлы |
|---|---|---|
| Заметки | `.services/notes/` | `notes.db`, `attachments/` |
| Финансы | `.services/finance/` | `finance.db` |
| Пароли | `.services/passwords/` | `vault.kdbx` |

## 6. Папка `.yeti`

Служебная папка самого Yeti Cloud. Не синхронизируется между устройствами — у каждого устройства своя.

| Файл | Назначение |
|---|---|
| `index.json` | Индекс всех файлов в папке: путь, SHA-256, размер, время изменения |
| `state.json` | Состояние синхронизации: last_sync_time по каждому устройству |
| `pending/` | Локальные изменения, ожидающие отправки на сервер |

## 7. Допустимые форматы и ограничения

| Параметр | Значение |
|---|---|
| **Максимальный размер файла** | Не ограничен (разбивается на чанки при передаче) |
| **Запрещённые символы в именах** | `< > : " / \ | ? *` (Windows-ограничения) |
| **Максимальная длина пути** | 260 символов (Windows-ограничение, планируется обход через `\\?\`) |
| **Рекомендуемые БД** | SQLite, LMDB (не требуют сервера, устойчивы к копированию) |
| **Запрещённые БД** | MySQL, PostgreSQL, MongoDB (требуют демон, не живут в файловой синхронизации) |

## 8. Частота синхронизации

| Устройство | Механизм | Интервал |
|---|---|---|
| **Windows (десктоп)** | `QFileSystemWatcher` + периодический опрос | Мгновенно при изменении + каждые 5 мин |
| **Android (телефон)** | WorkManager (периодическая задача) | Каждые 15 минут + опционально FCM push |
| **Сервер → офлайн-устройство** | При подключении устройства — сразу все pending-файлы | По факту подключения |
