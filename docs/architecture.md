# Yeti Cloud — архитектура и состояние проекта

## Общее описание

Yeti Cloud — self-hosted система синхронизации данных между устройствами. Состоит из трёх типов узлов:

| Узел | Роль | Язык | Статус |
|---|---|---|---|
| **Сервер (VPS)** | Координатор и транзитный узел | Go | Прототип |
| **Десктоп-клиент** | Центральный архив, может менять файлы | C++/Qt 6 | Прототип |
| **Android-клиент** | Главный узел, всегда онлайн | Java | Не начат |

## Структура проекта

| Каталог/Файл | Назначение |
|---|---|
| `desktop/` | Клиент Windows (C++/Qt 6) |
| `desktop/core/` | Ядро клиента: SSH-деплой, сетевое взаимодействие |
| `desktop/ui/` | UI клиента: окна, страницы, трей |
| `server/` | Сервер (Go): WebSocket-хаб, авторизация, HTTP API, конфигурация |
| `server/cmd/server/` | Точка входа Go-сервера |
| `server/internal/` | Внутренние пакеты: auth, config, hub, api |
| `server/internal/api/` | HTTP API: handler'ы, middleware, helpers |
| `docs/` | Документация проекта |
| `build/` | Сборочная директория CMake (в .gitignore) |
| `.vscode/` | Настройки VS Code: задачи, запуск, свойства C++ |

## Реализовано

### Клиент Windows (C++/Qt 6)
- Иконка в трее с меню: Показать, Стоп/Старт сервис, Выход
- Главное окно 360×600 в стиле Android
- Страница "Статус" с индикацией: онлайн/оффлайн, адрес сервера, наличие конфига, admin_key
- Страница "Добавить сервер": хост, порт, пользователь, пароль
- SSH-деплой через libssh:
  - Подключение и аутентификация по паролю
  - Загрузка бинарника сервера через SCP
  - Убийство старого процесса и запуск нового через nohup
- **Генерация `admin_key` (64 hex) при деплое**
- **Создание `yeti-desktop-config.json` в папке с exe-файлом** (поля: `admin_key`, `server_host`, `server_port`)
- **Автоподключение при старте** — если найден `yeti-desktop-config.json`, клиент автоматически соединяется с сервером
- **SCP-загрузка `yeti-server-config.json` на сервер в `/opt/yeti-server/`**
- WebSocket-подключение после деплоя с `?token=<admin_key>` в URL
- Индикация статуса: иконка онлайн/оффлайн, страница статуса
- **CMake-цель `yeti-server-build`**: автосборка Go сервера под Linux при F7

### Сервер (Go)
- **Загрузка/создание `yeti-server-config.json` при старте** (пакет `internal/config`)
- **Генерация `admin_key` при первом запуске**, сохранение в JSON
- **Валидация токена в WebSocket-хендшейке** (`?token=` из query)
  - Принимает `admin_key` (полный доступ) или валидный `device_token`
  - Невалидные токены — `401 Unauthorized`
- WebSocket-хаб: регистрация, отключение, broadcast сообщений
- Определение соседей за одним NAT (одинаковый публичный IP)
- Эндпоинт `/health`
- Структура токенов: admin, invite, device
- Хранение токенов в памяти

### HTTP API (новое)
- **`POST /api/invite`** — создание invite-токена (требует `Authorization: Bearer <admin_key>`)
- **`POST /api/register`** — регистрация устройства по invite-токену, возвращает `device_key` и `device_id`
- **`GET /api/devices`** — список зарегистрированных устройств с онлайн-статусом (требует `admin_key`)
- Middleware проверки `admin_key` в заголовке `Authorization`

### Инфраструктура
- Docker-контейнер `yeti-test-vps` с SSH (порт 2222) и сервером Yeti (порт 8080) для локального тестирования
- Сборка клиента: vcpkg + CMake + MSVC
- Сборка сервера: Go modules

## Следующий шаг (приоритетные задачи)

### 1. Регистрация десктоп-клиента как устройства [NEW]
После успешного деплоя сервера клиент должен:
- Создать invite-токен через `POST /api/invite` (используя `admin_key`)
- Зарегистрироваться через `POST /api/register` (используя invite-токен)
- Сохранить `device_key` и `device_id` в `yeti-desktop-config.json`
- Подключаться по WebSocket с `device_key`, а не с `admin_key`
- `admin_key` использовать только для вызовов HTTP API

### 2. Персистентность device-токенов в конфиге сервера
Сейчас `TokenStore` живёт в памяти — при рестарте сервера все device-токены теряются. Нужно расширить `yeti-server-config.json` полем `"devices": [...]` и сохранять/загружать их при старте/остановке.

### 3. Обработка WebSocket-сообщений на клиенте
Сейчас клиент подключается по WebSocket, но не обрабатывает входящие сообщения. Минимум — принять и отобразить `hello` от сервера. Это основа для дальнейшей синхронизации.

### 4. DPAPI для хранения ключа на клиенте
Сейчас `admin_key` лежит в `yeti-desktop-config.json` в открытом виде. На Windows можно зашифровать через `CryptProtectData` (DPAPI) и хранить в зашифрованном виде, расшифровывая только в рантайме.

## Протокол обмена (план)

### WebSocket (постоянное соединение)
Используется для:
- Регистрации устройства (hello)
- Обмена метаданными (индексы файлов)
- Координации прямых соединений

Типы сообщений:
- `hello` — регистрация устройства
- `peer_found` — найден сосед в локальной сети
- `peer_connected` — прямое соединение установлено
- `file_changed` — уведомление об изменении файла
- `file_available` — запрос на скачивание файла

### HTTP
Используется для:
- Управления токенами (invite, register)
- Передачи содержимого файлов, когда сервер в режиме транзита

## Токены

| Тип | Как появляется | Права | Срок |
|---|---|---|---|
| `admin_key` | Генерируется при деплое, сохраняется в конфиге | HTTP API: создание инвайтов, просмотр устройств | Бессрочный |
| `invite_token` | Админ создаёт через `POST /api/invite` | Только регистрация устройства | 24 ч, одноразовый |
| `device_key` | Выдаётся при регистрации через `POST /api/register` | WebSocket-синхронизация | 1 год |

**Важно:** `admin_key` и `device_key` разделены по назначению. `admin_key` — только для управления (HTTP API). `device_key` — для синхронизации (WebSocket). Десктоп-клиент админа использует `device_key` для подключения к WebSocket, как и любое другое устройство.

Сервер хранит только SHA-256 хеши device-ключей.

## Безопасность

- Ключи на клиентах: DPAPI (Windows), EncryptedSharedPreferences (Android)
- Invite-токены одноразовые
- Сервер не хранит ключи в открытом виде
- Админ может отозвать любое устройство
- `admin_key` не используется для WebSocket-подключений

## Система контроля версий

Используется удаленное хранилище гитхаб. Разработка ведется в ветке. Название ветки приблизительно такое - feature/yeti-cloud-0001 где 0001 порядковый номер ветки. Он может зависеть от номера задачи или просто меняться по мере добавления в Гитхаб.

Ветка считается завершенной, если хотя бы код компилируется и запускается, в нем нет ошибок. Либо есть ошибки, которые не останавливают и не вызывают аварийного сбоя программы или ОС. Ветка такая коммитится и помещается в ГитХаб.

Перед называнием ветки идут префиксы feature/ - новшества, улучшения. bugfix/ - устранение ошибок. (Возможны другие префиксы, может нейросеть предложит).

В Гитхабе, после помещения такой ветки, создается пулреквест и мержится с веткой master это основная ветка. Позже введем версии приложения. Это кстати одна из задач этого проекта.

Коммиты это промежуточные или конечные этапы выполнения работы в данной ветке. Они должны иметь комментарий типа yeti-cloud-0001 Init Yeti Cloud monorepo. Где указывается имя ветки без префикса и короткий комментарий, лучше на английском. В конце обязательно должно стоять точка.
