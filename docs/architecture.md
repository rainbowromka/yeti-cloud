# Yeti Cloud — архитектура и состояние проекта

## Общее описание

Yeti Cloud — self-hosted система синхронизации данных между устройствами. Состоит из трёх типов узлов:

| Узел | Роль | Язык | Статус |
|---|---|---|---|
| **Сервер (VPS)** | Координатор и транзитный узел | Go | Прототип |
| **Десктоп-клиент** | Центральный архив, может менять файлы | C++/Qt 6 | Прототип |
| **Android-клиент** | Главный узел, всегда онлайн | Java | Прототип |

## Структура проекта

| Каталог/Файл | Назначение |
|---|---|
| `android/` | Android-клиент (Java, Gradle) |
| `desktop/` | Клиент Windows (C++/Qt 6) |
| `desktop/core/` | Ядро клиента: SSH-деплой, сетевое взаимодействие |
| `desktop/ui/` | UI клиента: окна, страницы, трей |
| `server/` | Сервер (Go): WebSocket-хаб, авторизация, HTTP API, конфигурация |
| `server/cmd/server/` | Точка входа Go-сервера |
| `server/internal/` | Внутренние пакеты: auth, config, hub, api |
| `server/internal/api/` | HTTP API: handler'ы, middleware, helpers |
| `docs/` | Документация проекта |
| `build/` | Сборочная директория CMake (в .gitignore) |
| `.vscode/` | Настройки VS Code: задачи, запуск, свойства C++ |

## Реализовано

### Клиент Windows (C++/Qt 6)

#### Ядро (Client)
- **Клиент в отдельном потоке** (`Client` живёт в `QThread`, UI в главном потоке)
- **Авто-подключение при старте**: чтение `ConfigManager`, если есть `device_key` → WebSocket, если есть `admin_key` → HTTP-регистрация
- **HTTP-регистрация** (`HttpClient`): цепочка `POST /api/invite` → `POST /api/register`, получение `device_key` и `device_id`
- **WebSocket-клиент**:
  - Подключение с токеном `ws://host:port/ws?token=device_key&device_id=...`
  - Пинг/понг каждые 30 секунд (keep-alive)
  - Авто-реконнект при обрыве (5 попыток, задержка 5с → 10с)
  - Обработка входящих JSON-сообщений
  - Сигналы: `connected()`, `disconnected()`, `errorOccurred()`, `statusChanged()`
- **Логирование** (`LogBuffer`): кольцевой буфер на 50 строк, сигнал `newMessage` для UI

#### UI (MainWindow)
- Иконка в трее с меню: Показать, Стоп/Старт сервис, Выход
- Главное окно 360×600
- Страница "Статус": онлайн/оффлайн, адрес сервера, ключ
- Страница "Добавить сервер": хост, порт, пользователь, пароль, кнопка "Установить"
- **Консоль отладки** (`ConsoleWidget`): чёрный фон, зелёный текст, последние 4 строки из `LogBuffer`, кнопки Clear/Copy
- UI подписан на сигналы `Client` через `QueuedConnection`, не знает о сети напрямую

#### SSH-деплой (SshDeployer)
- Подключение и аутентификация по паролю (libssh)
- Загрузка бинарника сервера и конфига через SCP
- Убийство старого процесса и запуск нового через nohup
- Генерация `admin_key` (64 hex) при деплое
- Создание `yeti-server-config.json` и `yeti-desktop-config.json`
- **Все шаги логируются в `LogBuffer`**

### Android-клиент (Java)
- **UI**: Toolbar с заголовком, кнопка синхронизации, меню (Status, Deploy, Connect, Disconnect)
- **Фрагменты**: `StatusFragment` (статус соединения), `DeployFragment` (SSH-деплой)
- **SSH-деплой через JSch**:
  - Генерация `admin_key` (64 hex)
  - Загрузка бинарника сервера через base64 + exec
  - Создание конфигов на сервере и локально
  - Авто-подключение после деплоя
- **HTTP-регистрация** (`YetiClient`): OkHttp, цепочка `POST /api/invite` → `POST /api/register`
- **WebSocket-клиент**: OkHttp WebSocket, приём/отправка JSON-сообщений
- **Логирование**: консоль внизу экрана (40% высоты), зелёный текст на чёрном фоне
- **Хранение конфига**: `yeti-desktop-config.json` во внутреннем хранилище приложения

### Сервер (Go)
- **Загрузка/создание `yeti-server-config.json` при старте** (пакет `internal/config`)
- **Генерация `admin_key` при первом запуске**, сохранение в JSON
- **Валидация токена в WebSocket-хендшейке** (`?token=` из query)
  - Принимает `admin_key` (полный доступ) или валидный `device_key`
  - Невалидные токены — `401 Unauthorized`
- WebSocket-хаб: регистрация, отключение, broadcast текстовых сообщений
- Определение соседей за одним NAT (одинаковый публичный IP)
- Эндпоинт `/health`
- Структура токенов: admin, invite, device
- Хранение токенов в памяти (планируется персистентность)

### HTTP API
- **`POST /api/invite`** — создание invite-токена (требует `Authorization: Bearer <admin_key>`)
- **`POST /api/register`** — регистрация устройства по invite-токену, возвращает `device_key` и `device_id`
- **`GET /api/devices`** — список зарегистрированных устройств с онлайн-статусом (требует `admin_key`)
- Middleware проверки `admin_key` в заголовке `Authorization`

### Инфраструктура
- Docker-контейнер `yeti-test-vps` с SSH (порт 2222) и сервером Yeti (порт 8080) для локального тестирования
- Сборка клиента: vcpkg + CMake + MSVC
- Сборка сервера: Go modules

## Предстоит реализовать

### Авторизация и конфигурация
- **Персистентность device-токенов в `yeti-server-config.json`** — сейчас сервер сохраняет устройства, но при деплое старый конфиг перезаписывается. Нужно: перед деплоем скачивать существующий конфиг, сливать списки устройств, загружать обратно.
- **Имя устройства** — сейчас хардкод "Windows Desktop", нужно дать пользователю задавать имя.
- **Персистентность device-токенов в `yeti-server-config.json`** (сейчас токены теряются при рестарте сервера)
- **Удаление старого `yeti-server-config.json` при деплое** — перед загрузкой нового конфига на сервер нужно удалять старый, чтобы избежать конфликтов.
- **Политика редеплоя и восстановления сервера**:
  - При смене сервера (блокировка VPS, миграция) должна быть возможность восстановить работоспособность через любой admin-устройство.
  - Admin-устройство загружает на новый сервер `yeti-server-config.json` (со всеми device-ключами).
  - Подчинённым устройствам рассылаются новые инвайты или конфиги с адресом нового сервера.
  - **Крайний случай:** если admin-устройств нет, требуется механизм emergency recovery через seed-фразу или резервную копию конфига.

## Следующий шаг (приоритетные задачи)

### 1. Регистрация десктоп-клиента как устройства [NEW] [HIGH]
После успешного деплоя сервера клиент должен:
- Создать invite-токен через `POST /api/invite` (используя `admin_key`)
- Зарегистрироваться через `POST /api/register` (используя invite-токен)
- Получить `device_key`, `device_id` и **`device_name`** (имя устройства задаётся пользователем или генерируется из hostname)
- Сохранить `device_id` и зашифрованный `device_key` в `yeti-desktop-config.json`
- Подключаться по WebSocket с `device_key`, а не с `admin_key`
- `admin_key` использовать только для вызовов HTTP API

### 2. Персистентность device-токенов в конфиге сервера [MEDIUM]
Сейчас `TokenStore` живёт в памяти — при рестарте сервера все device-токены теряются. Нужно расширить `yeti-server-config.json` полем `"devices": [{"device_id": "...", "device_key_hash": "...", "device_name": "...", "created_at": "..."}]` и сохранять/загружать их при старте/остановке.

### 3. Удаление старого конфига при деплое [LOW]
При вызове `deployServer()` перед SCP-загрузкой `yeti-server-config.json` выполнять `rm -f /opt/yeti-server/yeti-server-config.json` через SSH. Это гарантирует чистый старт сервера без смешивания старых и новых ключей.

### 4. DPAPI уже реализован ✅
Шифрование ключей на клиенте работает через `CryptProtectData` / `CryptUnprotectData`.

## Протокол обмена (план)

### WebSocket (постоянное соединение)
Используется для:
- Регистрации устройства (hello)
- Обмена метаданными (индексы файлов)
- Координации прямых соединений

Типы сообщений (планируемые):
- `hello` — регистрация устройства (отправляется после connected)
- `peer_found` — найден сосед в локальной сети
- `peer_connected` — прямое соединение установлено
- `file_changed` — уведомление об изменении файла
- `file_available` — запрос на скачивание файла

### HTTP
Используется для:
- Управления токенами (invite, register)
- Передачи содержимого файлов, когда сервер в режиме транзита

## Токены

| Тип | Как появляется | Права | Срок |
|---|---|---|---|
| `admin_key` | Генерируется при деплое, сохраняется в конфиге | HTTP API: создание инвайтов, просмотр устройств | Бессрочный |
| `invite_token` | Админ создаёт через `POST /api/invite` | Только регистрация устройства | 24 ч, одноразовый |
| `device_key` | Выдаётся при регистрации через `POST /api/register` | WebSocket-синхронизация | 1 год |

**Важно:** `admin_key` и `device_key` разделены по назначению. `admin_key` — только для управления (HTTP API). `device_key` — для синхронизации (WebSocket). Десктоп-клиент админа использует `device_key` для подключения к WebSocket, как и любое другое устройство.

Сервер хранит только SHA-256 хеши device-ключей.

## Безопасность

- Ключи на клиентах: DPAPI (Windows), EncryptedSharedPreferences (Android — планируется)
- Invite-токены одноразовые
- Сервер не хранит ключи в открытом виде
- Админ может отозвать любое устройство
- `admin_key` не используется для WebSocket-подключений

## Система контроля версий

Используется удаленное хранилище гитхаб. Разработка ведется в ветке. Название ветки приблизительно такое - feature/yeti-cloud-0001 где 0001 порядковый номер ветки. Он может зависеть от номера задачи или просто меняться по мере добавления в Гитхаб.

Ветка считается завершенной, если хотя бы код компилируется и запускается, в нем нет ошибок. Либо есть ошибки, которые не останавливают и не вызывают аварийного сбоя программы или ОС. Ветка такая коммитится и помещается в ГитХаб.

Перед называнием ветки идут префиксы feature/ - новшества, улучшения. bugfix/ - устранение ошибок. (Возможны другие префиксы, может нейросеть предложит).

В Гитхабе, после помещения такой ветки, создается пулреквест и мержится с веткой master это основная ветка. Позже введем версии приложения. Это кстати одна из задач этого проекта.

Коммиты это промежуточные или конечные этапы выполнения работы в данной ветке. Они должны иметь комментарий типа yeti-cloud-0001 Init Yeti Cloud monorepo. Где указывается имя ветки без префикса и короткий комментарий, лучше на английском. В конце обязательно должно стоять точка.

**Текущая ветка:** `feature/yeti-cloud-0010` (клиент в отдельном потоке, рефакторинг HTTP-клиента, WebSocket пинг/понг, авто-реконнект, консоль отладки).
